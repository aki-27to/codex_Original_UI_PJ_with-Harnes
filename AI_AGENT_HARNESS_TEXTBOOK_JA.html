7. 主要フロー

7.1 Boot flow

  1. start_codex_ui.bat が UAC self-elevation を試みる。
  2. node と codex の存在を preflight する。
  3. CODEX_UI_PORT を数値検証し、無効なら 57525 に戻す。
  4. execution profile、request-user-input、dispatch guard、adversarial loop、git automation などの launcher defaults を未設定時のみ注入する。
  5. ブラウザ自動起動が有効なら、URL の応答を待って Edge もしくは system default browser を開く。
  6. node server.js を起動する。

7.2 Interactive exec flow

  1. browser UI が prompt / settings / images / workspace context を POST /api/exec へ送る。
  2. server が content-type、control token、agent name、workspace guard、idempotency key を検証する。
  3. design-sensitive web_ui source なら workspace lock と intent-first brief を強制する。
  4. app-server との initialize -> initialized handshake 後、JSONL stream で turn events を受ける。
  5. runtime は stream から command/file/mcp/collab/dispatch の observed signal を抽出する。
  6. turn 完了時に parent dispatch guard、intent-first gates、task outcome bridge、adversarial shadow review を評価する。
  7. turn artifact、latest turn snapshot、execution memory、replay baseline、operation log を更新する。
  8. taskOutcomeStatus=COMPLETED なら git automation が turn の cwd を対象に commit/push を試みる。

7.3 Replay / Eval flow

  1. replay は過去 turn の prompt と execution recipe を再構成し、必要なら repro profile で webSearch=0、forceNewSession=1、requestUserInputPolicy=blocked を強制する。
  2. eval は default suite core-harness-workflow.v4 を variants ごとに回し、exec case と deterministic probe case を混在実行する。
  3. eval report は logs/eval_runs.jsonl に append され、必要なら synthetic probe 結果を harness memory に保存する。

7.4 Conversation / Voice flow

  1. conversation request は same-origin 制約を通過した後、app-server provider へ送られる。
  2. persona_friend mode では request 前に persona memory を読み、request 後に message/topic をもとに更新する。
  3. TTS は Piper なら local model assets、Kokoro なら upstream HTTP で音声を生成する。

8. 制御境界と安全策

8.1 Control-plane 認証

Mechanism 現行挙動
validateControlMutationRequest() loopback same-origin と x-codex-control-token を両方要求。workspace/intention/open-cmd 系は action allowlist も要求。
validateLocalOriginRequest() conversation / voice 系で使用。token は不要だが same local origin は必須。
Open exception /api/runtime、/api/harness/overview、/api/diagnostics、/api/agent-topography、/api/eval/suites などは token 不要で読める。
Notable gap /api/batch/run、/api/batch/scheduler、/api/requirement-guard/validate は handler-level token check を持たない。

8.2 Workspace guard / intent-first gate

Workspace guard

  • scope は directory_tree。
  • locked root がある場合、cwd はその配下でないと reject。
  • web_ui source は unlocked だと preflight で reject。
  • folder picker は Windows のみ。

Design acceptance

  • benchmarkComparisonRequired=true
  • visualReviewRequired=true
  • independentReviewRequired=true
  • docSyncRequired=true
  • technicalVerificationRequired=true
  • required artifacts は locked intent summary、benchmark decomposition、desktop/mobile screenshot review、independent reviewer verdict、technical verification、documentation sync。

8.3 Requirement guard / RBJ / scope expansion

requirement guard extension point は scripts/extensions/requirement_guard_hook.js です。 prompt に marker を付与して requirement lock、scope expansion token、RBJ requirement definition loop を管理します。

  • RBJ の marker は [REQUIREMENT_RBJ_V1]、既定 parent agents は default / intake / release_manager。
  • RBJ 既定値は max_questions=3、max_revisions=2、min_confidence=80。
  • scope expansion は [SCOPE_EXPANSION_V1] と token 群で core-only / auto-enabled を切り替える。
  • requirement guard matcher は REQUIREMENT_GUARD_MATCH_VALUE と /api/requirement-guard/validate で疎通確認できる。

8.4 Parent dispatch guard / adversarial shadow

  • launcher default は CODEX_PARENT_DISPATCH_GUARD_MODE=enforce、CODEX_PARENT_DISPATCH_GUARD_MAX_RETRIES=1。
  • parent が child dispatch なしで completion しようとすると、warn では警告、enforce では final status を block できる。
  • adversarial shadow は既定 enabled。minScore=72、prompt max 8000 chars、answer max 16000 chars。
  • adversarial loop は既定 enabled、maxRetries=1。FAILED_VALIDATION などを retry 根拠に使う。

8.5 Idempotency lifecycle

  • key は header idempotency-key または body idempotencyKey。両方あり不一致なら 400。
  • 同一 key で request hash が違うと 409 idempotency_request_hash_mismatch。
  • running duplicate は 409。resolved duplicate は保存済み outcome snapshot を 200 で返す。
  • snapshot は lifecycle running / terminal states / released を持つ。
  • 既定 TTL は 30 min、status wait 上限は 30000 ms。

8.6 Git automation

  • turn-complete で走る。対象は taskOutcomeStatus=COMPLETED の turn のみ。
  • 対象 repo はその turn の cwd。baseline が dirty なら skip。
  • remote 未設定または detached HEAD なら push しない。
  • この harness repo に対しては logs/harness_execution_memory.json と logs/eval_runs.jsonl を baseline 判定から除外する。

13. 教科書補遺: 正本 hierarchy と読み解き順

この repo は docs だけを順に読めば自動的に全部理解できる構造ではありません。 実務では、次の順に真偽を詰めると誤読しにくくなります。

  1. AGENTS.md: Tier-0 憲法。identity、success、hard constraints、completion、approval boundary の最上位 rule。
  2. docs/AGENT_OPERATING_RULES.md と関連 policy docs: parent/child flow、skill routing、evidence gate の運用ルール。
  3. docs/CURRENT_ARCHITECTURE.md: 現在有効な設計仕様。
  4. scripts/config/*.json: machine-readable contracts。enum や field はここを正本として読む。
  5. server.js と logs/*: 実 runtime truth。仕様との差異確認もここで行う。

文書間にズレがある場合は「憲法 → contract → runtime observation」の順で優先度をつけます。 この順番を外すと、文字化けした README や更新途中の changelog に引っ張られて解釈を誤ります。

14. 教科書補遺: agent / skill / task family の要点

Parent roles

  • default: 唯一の general-purpose parent。orchestration、dispatch、final review、final report を担当。
  • intake: Step 1/2 専用 planner。read-only。
  • release_manager: Step 4/5 専用 gate。read-only。

Child roles

  • frontend_worker: web/ UI/UX、browser behavior。
  • backend_worker: server.js、scripts/、protocol/API。
  • infra_worker: .codex/、logging、launcher、runtime config。
  • tester: executable verification。
  • reviewer: read-only independent review。
  • explorer: read-only fact finding。

主要 skill assignment

  • default: spec-sync-assistant、parent-dispatch-guard、red-requirement-auditor など。
  • release_manager: release-evidence-gate、turn-log-auditor など。
  • frontend_worker: playwright、screenshot、ui-regression-diff。
  • backend_worker: appserver-protocol-debugger、api-contract-testgen など。

task family

  • deterministic_code: correctness first。最低 planning は FAST。
  • web_creative: wow first。最低 planning は NORMAL。completion contract は design acceptance。
  • research_analysis: coverage and insight。
  • planning_design: decision support。

親が skill を知っていても specialist workflow を自分で代行してよいわけではありません。 skill は capability の札であり、parent の免罪符ではありません。

15. 教科書補遺: adaptive execution の最短理解

Planning modes

  • FAST: 既存スコープ、小さな変更、open question がほぼない。
  • NORMAL: 複数 specialist が絡むが、実行可能な範囲に収まる。
  • DISCOVERY: 承認境界や open question が多く、実装を止めるべき状態。

Assurance depth

  • LIGHT_ASSURANCE: docs-only など低リスク。
  • STANDARD_ASSURANCE: 通常実装。
  • SIGNOFF_ASSURANCE: runtime、protocol、governance、infra、新ロジック。

Intent-first design gate

design-sensitive work では benchmark、taste memory、desktop/mobile screenshot、independent reviewer、doc sync が欠けると COMPLETED ではなく FAILED_VALIDATION または NEEDS_INPUT と扱うのが基本です。
