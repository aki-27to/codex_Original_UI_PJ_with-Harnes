音声実装比較

音声会話実装ガイド（品質・安定性・無料制約）

結論として、音質・安定性・低遅延を最優先するなら OpenAI Realtime + gpt-4o-transcribe + gpt-4o-mini-tts + WebRTC が最有力です。 一方、完全無料を厳守するならローカル完結構成が現実解です。

候補比較（要点）

候補 音質/認識品質 安定性 遅延 声選択 コスト
OpenAI Realtime / Speech API 高い 高い（VAD/ノイズ低減設定あり） 低遅延（WebRTC向き） 可能（Realtime: 10声 + custom voice、TTS: 13声） 従量課金
Browser Web Speech API 中〜高（環境依存） 中（特に認識はブラウザ差が大きい） 中 可能（OS/ブラウザ提供声） 追加費用ほぼ不要
ローカル構成（whisper.cpp + LLM + Piper） 中〜高（モデル/GPU次第） 高い（オフライン運用可） 中（マシンスペック依存） 可能（Piperモデル選択、多話者モデルはspeaker指定） ソフト無料（電力/ハードは必要）
ローカル軽量STT（Vosk） 中〜高（用途次第） 高い（軽量で安定） 低〜中 STTなので声選択は対象外 無料

候補別の特徴

1. OpenAI Realtime / Speech API

  • 音声入出力と会話制御を1つの仕様で扱えるため、運用安定性が高い。
  • Realtimeの音声は alloy / ash / ballad / coral / echo / sage / shimmer / verse / marin / cedar が利用可能。
  • TTSガイドでは13音声が提示され、音質面では marin / cedar 推奨。
  • Realtimeでは初回音声出力後にvoice変更制約があるため、セッション開始時にvoice固定が安全。

2. Browser Web Speech API

  • speechSynthesis.getVoices() で利用可能な声を列挙して選択できる。
  • 認識API（SpeechRecognition）は対応ブラウザが限定されるため、安定性に差が出る。
  • 低コストで導入しやすいが、品質や挙動の一貫性はクラウドAPIより弱い。

3. ローカル完結構成（完全無料寄り）

  • STT: whisper.cpp（高品質寄り）または Vosk（軽量寄り）。
  • LLM: llama.cpp または Ollama。
  • TTS: Piper（モデルによって音質と話者バリエーションが変わる）。
  • クラウド課金を避けられる一方で、マシンスペック依存が強い。

注意: Piper周辺はリポジトリ/ライセンスの変遷があるため、 採用時は利用するモデルの配布条件を都度確認してください。

声は選べるか？（結論）

技術 声選択 補足
OpenAI Realtime 可能 ビルトイン声に加え、仕様上custom voice idも扱える。
OpenAI TTS 可能 モデルにより利用可能音声セットが異なる。
Browser speechSynthesis 可能 OS/ブラウザ依存。端末にある声のみ利用可能。
Piper 可能 インストール済み音声モデル次第。多話者モデルはspeaker指定可能。
whisper.cpp / Vosk 不可（対象外） STTなので「声を出す」機能は持たない。

最も高パフォーマンスを狙う実装方針

  • 転送はブラウザなら WebRTC を優先。
  • 入力環境に応じて near_field / far_field ノイズ低減を切り替える。
  • ターン検出は対話用途なら server_vad or semantic_vad を検証して最適化。
  • STTは gpt-4o-transcribe を優先し、必要時のみ軽量モデルへ切替。
  • TTSはまず marin / cedar を基準音声にし、UX検証後に選択UIを公開。
  • セッション開始時にvoiceを確定し、途中変更制約による不整合を回避。
