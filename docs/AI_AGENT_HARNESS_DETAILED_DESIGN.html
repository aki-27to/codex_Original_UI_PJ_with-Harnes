Overview First

AI Agent Harness のしくみを、やさしい順番で説明する本

この文書は、AI Agent Harness を「何のための仕組みか」から順に理解するための解説書です。 先に数や一覧を並べるのではなく、まず全体像、その次に役割、最後に細かい仕組みへ進みます。

小学生にも読めるように短い文で書きます。ただし内容は今の実装に合わせます。 つまり、やさしいけれど、ふわっとした紹介ではなく、実際の動きに沿った説明です。

これは何?

ブラウザで受けたお願いを、親エージェントと専門エージェントのチームで安全に進める仕組みです。

Big Picture

1. まずは全体像

Harness は、AI に仕事をさせるための「受付つき作業チーム」です。 ただ AI に丸投げするのではなく、誰が何をしたかを見えるようにしながら進めます。

ひとことで言うと

人がお願いを出す。

親エージェントが「何をすれば成功か」を先に決める。

そのあと、合う専門家に仕事を任せる。

最後に、本当に終わったかを証拠つきで確かめる。

この仕組みが守りたいもの

1つ目は、ユーザーの意図どおりに動くことです。

2つ目は、勝手に仕事を広げすぎないことです。

3つ目は、「できました」と言う前に証拠をそろえることです。

大事な見方

この Harness の中心は、画面の数でも、API の数でもありません。 「お願いがどう流れて、どこで安全確認が入り、どこに証拠が残るか」が中心です。

Cast

2. だれが何をするのか

しくみを理解するときは、まず登場人物を知ると読みやすくなります。 ここではやさしい呼び方と、実際の名前をセットで書きます。

お願いする人

人間の利用者です。ブラウザの画面から依頼を書きます。

入口の画面は主に web/01.HarnesUI/ です。

受付と交通整理

ローカルサーバーの server.js です。

依頼を受け取り、実行の道筋と記録の置き場を整えます。

まとめ役

親エージェントです。ふつうは default が動きます。

「まず何を決めるか」「誰に任せるか」を決める役目です。

専門役

frontend_worker、backend_worker、infra_worker などです。

画面、API、設定、検証のように、得意分野ごとに担当します。

見張り役

reviewer と tester です。

必要なときに、別目線で確かめてバグや抜けを減らします。

記録係

logs/ です。

「何をしたか」「なぜそう判断したか」を後から追えるように残します。

補足

この文書では Parent を「まとめ役」、Child を「専門役」と呼びます。 親エージェントには default のほかに、入口整理に寄る intake と、 最終判定に寄る release_manager があります。 でも日常の主役はまず default です。

Journey

3. 1回の依頼は、どう進むのか

ここがこの文書の中心です。 細かい一覧を見る前に、まず「1回の依頼の旅」を頭に入れてください。

1

人が依頼を書く

利用者はブラウザの画面で依頼を書きます。画像や設定もここで一緒に送れます。

主な送り先は POST /api/exec です。

2

サーバーが受け取って、今の状態をそろえる

サーバーは依頼を受け取り、必要ならセッションを整えます。

同じ依頼が事故で二重送信されても壊れにくいように、再送対策も持ちます。

3

まとめ役が「成功の形」を先に決める

これを intent-first と考えるとわかりやすいです。先に「何がゴールか」「何をしないか」を固定します。

つまり、作業前に約束を作る段階です。

4

合う専門役に仕事を渡す

画面の仕事なら frontend_worker、API なら backend_worker のように振り分けます。

親が全部ひとりで片づけるのではなく、専門家に任せるのが基本です。

5

必要なレビューと検証をする

動いたかだけでは足りません。

バグがないか、ユーザー意図から外れていないか、証拠がそろっているかを見ます。

6

文書と記録をそろえて、最後の判定を出す

必要な文書反映まで終えたら、Harness は最終のリリース判定を出します。

このとき、結果だけでなく、残っている前提や注意点も一緒に報告します。

ここで覚える一番大事なこと

この Harness は「すぐ答える仕組み」より、「意図を固定して、専門家に任せて、証拠を残して終わる仕組み」です。 だから、旅の途中に確認の段階がいくつか入ります。

Safety

4. なぜ安全ルールがいるのか

速さだけを求めると、たいていどこかで困ります。 この Harness は、よくある失敗を先回りして減らすように作られています。

勝手に話を広げない

ユーザーが求めていない大きな改造を、黙って足さないようにします。

まず baseline を守り、その近くで小さな付加価値だけを許します。

まとめ役が専門役のふりをしない

親エージェントは司令塔です。実装の主役を勝手に奪わないように guard が入ります。

これで役割分担が崩れにくくなります。

証拠なしで完了と言わない

「できました」だけでは不十分です。

検証結果、レビュー、文書反映、残るリスクまで見て初めて完了になります。

やさしい言い換え

intent-first は「作る前に約束を決めること」です。 evidence は「ほんとうにやったとわかる記録」です。 release decision は「最終的に出してよいかの判定」です。

Surfaces

5. どの画面が何の役目を持つのか

画面はたくさんの情報を見せるためにあります。 でも、画面そのものが主役ではありません。主役はその後ろの流れです。

Console

依頼を出す主画面です。今の作業、計画、判定、進み具合を見る場所でもあります。

Overview

全体の状態を見る画面です。runtime、契約、証拠、履歴をまとめて見たいときに向きます。

Guide

guide.html は静的な説明ページです。動いている最中のデータに頼らず、考え方を説明します。

境界の話

付帯アプリも同じサーバーから見えることがありますが、この文書で中心にするのは Harness 自身のしくみです。 つまり、「依頼を受けて安全に進める本体」の話を優先します。

Records

6. 記録はどこに残るのか

記録の場所を知ると、トラブル調査もレビューも急に楽になります。 この Harness は記録の置き場を3つの層に分けています。

logs/current/

今すぐ見るための場所です。いちばん最初に見るならここです。

特に operator_summary.json は「いま何が起きているか」をつかむ入口です。

logs/bundles/

証拠をひとまとまりにした場所です。

proof や signoff のように、後で提出したり見返したりする単位で残します。

logs/archive/raw/

いちばん細かい生データです。

普段は深追いしなくてよいですが、何が本当に起きたかを厳密にたどるときに使います。

覚え方

current は今日のノート、bundles は提出用のファイル、 archive/raw は倉庫の原本だと思うと整理しやすいです。

Troubleshooting

7. 困ったときは、どこから見るべきか

「なんだか動かない」と感じたら、全部を一気に見る必要はありません。 次の順で見れば、だいたい迷子になりません。

  1. GET /api/runtime が 200 で返るかを見る。まず本体が起きているかを確かめます。
  2. Console の Harness Status を見る。今の作業段階と止まり位置がわかります。
  3. logs/current/operator_summary.json を見る。人向けの要約はここが入口です。
  4. もっと深く知りたいなら logs/bundles/ と logs/archive/raw/ へ降ります。
  5. 運用ルールを確認したいなら docs/APP_SERVER_PROTOCOL_RUNBOOK.md と docs/EVIDENCE_CONTRACT.md を開きます。
よくある勘違い

画面に答えが出ていないからといって、すぐに全部が壊れたとは限りません。 途中で review 待ち、evidence 不足、doc sync 未完了のように、 「安全のために止まっているだけ」の場合もあります。

Deeper View

8. もう少し詳しい仕組み

ここからは、全体像をつかんだ人向けの少し詳しい話です。 先にここから読むより、前の章を読んでから戻るほうが理解しやすいです。

主な入口は 2 つ

POST /api/exec は仕事を始める入口です。

GET /api/runtime は今の状態を読む入口です。

この 2 つを押さえるだけでも、Harness の骨組みはかなり見えます。

ローカルファースト

UI はローカル中心で動き、既定ポートは 57525 です。

つまり、まず手元で完結しやすい形を大事にしています。

再送に強くする工夫

同じ依頼が二重に走って混乱しないように、idempotency key を使います。

やさしく言うと「同じ伝票を見分ける番号」です。

最終判定は generic completion ではない

上位の判定は RELEASE_APPROVED などの business release state で終わります。

単なる COMPLETED だけではなく、出してよいかまで見る最終ラベルです。

人向け説明より強いもの

この文書は理解の助けです。

でも、厳密な正本は machine-readable contracts と runtime proof です。つまり、機械が読める約束と実行の証拠のほうが強い、という意味です。

付加価値の条件

よい追加提案でも、ユーザー意図を変える大きな寄り道は許しません。

小さく安全で、専用の証拠があるときだけ over-delivery を許します。
