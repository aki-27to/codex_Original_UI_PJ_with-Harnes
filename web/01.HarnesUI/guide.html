Child-Friendly System Book

これは broad runtime shell ではなく、governed autonomous worker platform を一枚で理解するための入口です。

価値の中心は「AI が done と言ったが出せない」を減らすこと、 「長時間タスクの引き継ぎ debt」を下げること、 そして「ship / no-ship を evidence で正直に決める」ことです。 標準ルートは POST /api/exec、評価と release judgment は POST /api/eval/run に固定されています。

実行は POST /api/exec 評価は POST /api/eval/run 証拠は logs/current/ と bundle に残る parallel harness は前提にしない

One Minute

最初にこれだけ読めばよい

能力名ではなく、実際に任せられる 3 つの仕事で breadth を見せます。

Delegated implementation

実装を進め、必要なら specialist を使い、最後は証拠つきで終わらせる仕事です。

Governed review / release decision

レビュー負担を軽くしつつ、ship / no-ship を hope ではなく evidence で判断する仕事です。

Long-horizon continuity / bounded improvement

セッションをまたいで文脈と残課題を持ち越し、改善は bounded に進める仕事です。

Agents

だれが何を担当するか

親がまとめ、必要に応じて専門子が実行します。

親の役割

  • default: 入口、計画、委譲、統合、最終報告
  • intake: 要件の固定と方向の整理
  • release_manager: evidence を見て release judgment を出す役

専門子の役割

  • frontend_worker: UI / UX
  • backend_worker: server / API
  • infra_worker: 環境 / 運用
  • tester: 検証
  • reviewer: 独立レビュー
  • explorer: 調査

ここで大事なこと

これは judge 専用の箱ではありません。親が worker の仕事をまとめ、必要な証拠を集め、最後に honest に判断する構造です。

Flow

仕事はどう進むか

難しい言葉より、進み方で覚えると分かりやすいです。

1. Requirement

何を達成したいか、何をやらないか、何を満たせばよいかを固定します。

2. Planning

どこまで自律で進め、どこで specialist を使うかを決めます。

3. Execution

必要な作業を進めます。長い仕事なら continuity を残します。

4. Quality Check

テスト、review、evidence で false complete を減らします。

5. Report

何が変わったか、何を確認したか、何が残るかを正直に報告します。

実装の見え方

コード変更、テスト、レビュー、ログの 4 つがそろって初めて「終わった」と言えます。

継続の見え方

長い仕事は次に何をやるかまで含めて残します。途中で session が変わっても guesswork を増やしません。

APIs

どんな API があるか

全部覚える必要はありません。中心だけ押さえれば十分です。

メインの API

  • POST /api/exec: 実行の中心
  • GET /api/runtime: 今の runtime を見る
  • GET /api/harness/overview: 俯瞰情報をまとめて取る

評価と replay の API

  • GET /api/eval/suites
  • GET /api/eval/history
  • POST /api/eval/run: 評価と release judgment の中心
  • GET /api/replay/turns
  • POST /api/replay/turn

会話と補助の API

  • POST /api/conversation/direct
  • GET /api/conversation/runtime
  • GET /api/slo/status

補助ルート

  • POST /api/voice/piper / POST /api/voice/kokoro
  • /api/batch/*: ローカル補助
  • POST /api/open-cmd
  • POST /api/requirement-guard/validate

いちばん大事なこと

標準ルートは POST /api/exec と POST /api/eval/run です。新しい主役ルートを増やして breadth を見せる設計ではありません。

Extra Rooms

追加の部屋

主役ではないけれど、知っていると便利な部屋です。

English Conversation App

same-origin の統合アプリとして使えます。必要なら standalone launcher を 127.0.0.1:57526 で起動し、conversation / TTS だけ main harness に proxy します。

  • browser path は /english-conversation-app/index.html
  • same-origin でも standalone でも main harness の実行面とは分離
  • conversation / voice だけを軽く試したいときに向いています

音声の補助

Piper と Kokoro を使った補助ルートがあります。主役の価値ではなく、体験の補助です。

  • /api/voice/piper
  • /api/voice/kokoro

Git 自動化

ローカル運用では commit / push を補助できますが、これを product identity の中心には置きません。

  • dirty baseline を無視してよいという意味ではありません
  • runtime state だけで publish 判断をしません
  • 常に evidence と release judgment を見ます

Overview との違い

Overview は live な俯瞰です。Guide は比較軸、意味、使い分けを固定文で説明する静的な front surface です。

Logs

あとで何を見ればよいか

このハーネスの強みは、説明ではなく事実が残ることです。

logs/current/

最新の要約を見る場所です。まずは operator_summary.json を見ます。

logs/bundles/

proof と signoff のまとまりを見る場所です。

logs/archive/

過去の実行を追いかける場所です。通常は毎回ここまで見る必要はありません。

最初に見るおすすめ順

  1. logs/current/operator_summary.json
  2. logs/current/latest_run_summary.json
  3. logs/current/latest_signoff_summary.json
  4. 必要なら bundle の raw を見る

なぜこれが大事か

「終わったらしい」ではなく、「何が起きたか」をあとから辿れるからです。review trust burden をここで下げます。

Safety

どんなガードがあるか

止めるためのガードではなく、危ない方向に quietly drift しないためのガードです。

いつもの前提

  • local-first で動く
  • 標準ポートは 57525
  • 不要な dependency 追加を主役にしない
  • 標準実行ルートは POST /api/exec を守る

入力と承認の姿勢

  • In the live runtime request-user-input uses auto-default so low-risk clarification can continue without stopping.
  • In the strict lane blocked stays in force for conversation, proof, and repro work.
  • 承認境界は監査メモであって、毎回の停止トリガではありません。
  • 無言の scope expansion と silent rewrite を禁止します。
  • 未検証の completion claim を禁止します。

子ども向けに言い換えると

「勝手に別の宿題に変えない」「できたふりをしない」「危ないことは quietly 通さない」というルールです。

Limits

まだ限界もある

よくできていても、万能ではありません。残る限界は最初から見せます。

実装核はまだ重い

server.js と web/01.HarnesUI/app.js の中心核はまだ太く、理解コストの大きい場所が残っています。

breadth は派手さで見せていない

browser、subagents、memory、continuity はありますが、派手な runtime shell の見え方で競っていません。

public landing そのものではない

この Guide は静的な front surface ですが、公開 landing page や 1-command install page の代わりではありません。

説明書と live の差

このページが正しくても、実際の runtime や logs を見ないと最新状態は分かりません。

Glossary

よく出ることば

意味が分かると、画面もログも読みやすくなります。

Harness

仕事を受けて、進めて、確かめて、判断して、残すための仕組み全体です。

Runtime

いま動いている現在の状態です。

Evidence

「ちゃんとやった」と言うための事実です。ログ、テスト、レビュー結果などを含みます。

Signoff

最後の確認です。出してよいかを honest に判断します。

Replay

前の実行をもう一度たどって見ることです。

Residual Risk

がんばっても、まだ残る前提や不確実さのことです。

Takeaway

最後にひとことで

このページを一文で言い直すとこうなります。

このハーネスは、Codex に仕事を任せるだけの箱ではありません。 delegated implementation、governed review / release decision、long-horizon continuity / bounded improvement を local-first にまとめ、最後は証拠と honest な judgment で閉じる governed autonomous worker platform です。
