Child-Friendly System Book

このハーネスは、標準入口を守りながら「受ける」「分ける」「証拠で閉じる」を行う司令室です。

むずかしい言い方をやめると、この仕組みは「先生ロボットの教室」です。 本線は POST /api/exec と POST /api/eval/run のままで、 役わり分担、契約、証拠、fail-closed な評価を上から重ねています。

入口は `POST /api/exec` eval は `POST /api/eval/run` 証拠は `logs/current` と bundle に残る parallel harness は増やさない

One Minute

まず1分でわかる説明

最初にここだけ読めば、このハーネスの大きな役目がつかめます。

1. 何をする場所?

人が書いたお願いを受けて、Codex に仕事をさせる場所です。本線は POST /api/exec、評価は POST /api/eval/run です。

2. どうやって安全にする?

いきなり全部を勝手にやらず、要件を固定し、親と子で役わりを分け、最後に証拠と release judgment を残します。

3. 何が残る?

今のまとめ、証拠セット、深い生ログの3段階で記録が残ります。`agi_v1` では fail-closed な評価と昇格判定も残ります。

Screens

どの画面が何をするの?

使い分けを知ると、迷いにくくなります。

Console

仕事を投げるメイン画面です。今の作業、計画、実行トレース、ステータスを見ます。

「今まさに何をしているか」を見る場所です。

Overview

全体の健康診断ページです。Runtime、Topology、Contracts、Evidence、Memory をまとめて見ます。

「教室全体がちゃんとしているか」を見る場所です。

Guide

このページです。画面の見方、役の意味、ログの探し方を、人に説明できる言葉でまとめています。

「そもそもこれは何か」を知る場所です。

Agents

だれが何をするの?

人のチームにたとえると、親と専門家に分かれています。

親役

  • default: ふつうの親。全体を見て、専門家にお願いして、最後にまとめます。
  • intake: 最初だけを見る親。何を求められているかを固めます。
  • release_manager: 最後だけを見る親。証拠が足りるかを見て、出してよいか決めます。

専門の子役

  • frontend_worker: 画面まわり担当
  • backend_worker: サーバーや API 担当
  • infra_worker: 起動、設定、運用担当
  • tester: テスト担当
  • reviewer: バグ探し担当
  • explorer: 調査担当

もう現役ではないもの

worker は昔の名前で、今の実行先としては使いません。いま残っているのは「古い指定を見つけて、だめだと教えるための痕跡」です。

Flow

仕事はどう流れるの?

「お願いして終わり」ではなく、5段階で進みます。

1. Requirement

やること、やらないこと、守る条件を決めます。

2. Planning

どんな順番で進めるか、だれに任せるかを決めます。

3. Execution

必要な役が実際に仕事をします。

4. Quality Check

テストやレビューで、ほんとうに大丈夫か確認します。

5. Report

結果、前提、残る心配ごとをまとめて報告します。

計画の深さ

軽い仕事は軽く、重い仕事は深く考えます。だから全部を同じ重さで扱いません。

確認の深さ

小さい直しと本番前の確認は同じではありません。必要に応じて review や signoff まで進みます。

APIs

どんな API があるの?

全部覚えなくて大丈夫です。大きな役目ごとに見るとわかりやすいです。

メインの API

  • POST /api/exec: 仕事を実行する正面入口
  • GET /api/runtime: 今の姿をみる
  • GET /api/harness/overview: 俯瞰情報をまとめて取る

会話と確認の API

  • POST /api/conversation/direct: 会話系の入口
  • GET /api/conversation/runtime: 会話の runtime 情報
  • GET /api/slo/status: 健康状態を見る

評価と再生の API

  • GET /api/eval/suites
  • GET /api/eval/history
  • POST /api/eval/run: 評価の本線です。agi_v1 もこの既存ルートで有効にします。
  • GET /api/replay/turns
  • GET /api/replay/turn/:turnId
  • POST /api/replay/turn

補助の API

  • POST /api/voice/piper と POST /api/voice/kokoro: 音声用
  • /api/batch/*: まとめ処理用
  • POST /api/open-cmd: 補助操作
  • POST /api/requirement-guard/validate: 条件の見張り番

いちばん大事なこと

API はたくさんありますが、仕事の本線は POST /api/exec、評価の本線は POST /api/eval/run です。補助 API はありますが、本線の代わりにしてはいけません。

Extra Rooms

このハーネスには別の部屋もあるの?

メインの仕事場のほかに、横につながっている部屋があります。

English Conversation App

同じサーバーの中にぶら下がる英会話アプリです。別ポートではなく、同じ場所から開けるので連携しやすくなっています。

  • URL は /english-conversation-app/index.html
  • 置き場の優先順は env 指定 → sibling repo → bundled fallback
  • 会話 API や TTS と同じ origin を使います

音声の部屋

声を出すために Piper や Kokoro を使えます。これは本体の仕事実行とは別の補助機能です。

  • /api/voice/piper
  • /api/voice/kokoro

Git 自動化

このハーネスは、完了した作業を自動で commit と push まで進める設定を持てます。

  • ただし dirty baseline なら勝手に公開しません
  • runtime state だけの変化では publish を起こしにくくしています
  • つまり「便利だけど、むやみに危なくしない」考えです

Overview と違うところ

Overview は「いま」の数字と状態を見せる画面です。ここは「何のためにあるか」を言葉で教える画面です。

Logs

あとで何を見ればいいの?

このハーネスの強みは、証拠がちゃんと残ることです。

logs/current/

今すぐ読む場所です。まずは operator_summary.json から見ます。

logs/bundles/

証拠セットの箱です。proof や signoff のまとまりが入ります。

logs/archive/

深い生ログ置き場です。ふだんはここまで見なくても大丈夫です。

最初に見るおすすめ順

  1. logs/current/operator_summary.json
  2. logs/current/latest_run_summary.json
  3. logs/current/latest_signoff_summary.json
  4. 必要なら bundle の中や raw を見る

どうして3段階に分かれているの?

いつも全部の生ログを見ると大変だからです。まずやさしいまとめを見て、必要なときだけ深く潜る作りになっています。

Safety

どんな安全ルールがあるの?

ただ動けばよいのではなく、勝手に危ないことをしにくい作りです。

いつもの約束

  • local-first で動く
  • 標準ポートは 57525
  • 追加 dependency を勝手に増やさない
  • 標準実行 path は POST /api/exec を守る

ガードレール

  • request-user-input は既定で auto-default
  • 親は専門作業を自分で抱え込みすぎない
  • 証拠なしで「終わった」と言わない
  • 勝手な scope expansion をしない

小学生向けのひとこと

「近道に見える裏口」があっても、先生は正門から入るべき、というルールです。そのほうがあとで困りません。

Limits

今も気をつけること

よくできていますが、万能ではありません。残る注意点も知っておくべきです。

外の英会話アプリ

sibling repo を優先して読めるので便利ですが、その中身が API と合っているかは別に確認が必要です。

stdio の実機確認

ホストが子プロセス起動を止める環境だと、完全な live 確認ができないことがあります。

Glossary

むずかしい言葉をやさしく

意味がわかると、文書を追いやすくなります。

Harness

仕事の流れを安全に固定して、途中と最後を見えるようにする仕組み。

Runtime

いまこの瞬間に動いている本当の状態。

Evidence

「ちゃんとやった」と言うための証拠。ログ、テスト結果、レビュー結果など。

Signoff

最後の確認。学校でいうと「提出前の最終チェック」です。

Replay

前の動きをもう一度たどって見ること。

Residual Risk

直したあとでも、まだ少し残る心配ごと。

Takeaway

最後にひとことで

だれかに説明するときは、この一段落で十分です。

このハーネスは、Codex に仕事をさせるための「安全な司令室」です。 仕事の入口はひとつで、親役が計画して、専門の子役が作業し、最後はレビューやテストの証拠を残します。 Console は作業場、Overview は全体地図、Guide はやさしい説明書です。 迷ったら、まず logs/current/operator_summary.json と Overview を見れば、いま何が起きているかを追いやすくなります。
