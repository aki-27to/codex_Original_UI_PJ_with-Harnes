# review-agent-essence

対象のスキル・コード・設計文書を `reference/agent-essence.md` の原則に照らしてレビューし、設計上の強み・弱み・改善案を返す。

**このスキルは対象を直接修正しない。**

## Input / Output

- **Input**: `$ARGUMENTS` = 対象のパス（スキル、コード、設計文書）
- **Output**: 標準出力にフィードバックを返す
- **評価基準**: `reference/agent-essence.md`

## 手順

### 1. 読み込み

1. `reference/agent-essence.md` を読み、全原則を把握する
2. `$ARGUMENTS` の対象を読む（SKILL.md + 補助ファイル、またはコード/設計文書）

### 2. 関連原則の選定

対象の性質に応じて、11原則のうち関連するものを選定する。全原則を機械的に当てはめない。

選定基準:
- その原則が対象の設計判断に実際に影響するか
- 違反した場合に実害が生じるか
- 適用することで具体的な改善が見込めるか

### 3. 評価 → フィードバック出力

以下のフォーマットで出力する:

```markdown
# Essence Review: {対象名}

> 評価基準: [agent-essence](reference/agent-essence.md)

## 対象の要約

{対象が何をするものか、1-2文で}

## 原則適用マトリクス

| # | 原則 | 関連度 | 判定 | 根拠 |
|---|------|--------|------|------|
| 1 | 関心ごとの分離 | {高/中/低/-} | {○/△/×/-} | {1文で} |
| 1.1 | コンテキストはゼロサム | ... | ... | ... |
| 1.2 | 入力バイアスの増幅 | ... | ... | ... |
| 2 | 記憶の外部化 | ... | ... | ... |
| ... | ... | ... | ... | ... |

※ 関連度「-」= 対象に無関係、判定も「-」にする

## 主要な指摘

### 強み
{原則に沿った良い設計判断を具体的に挙げる}

### 改善提案
{関連度「高」で判定「△」「×」のものについて、具体的な改善案を提示}

各提案は以下の形式:
- **原則 #N.M {原則名}**: {現状の問題} → {具体的な改善案}

### 見落としリスク
{対象が暗黙に前提としているが、原則に照らすと危うい設計判断があれば指摘}

## 総評

{2-3文で要約。対象の設計成熟度と最優先の改善点}
```

## Gotchas

- 全原則を均等に扱わない。対象に無関係な原則は「-」にして飛ばす。無理に当てはめるとノイズになる
- 「原則に書いてないから問題ない」は誤り。原則は網羅的チェックリストではなく思考の補助線
- 判定「○」にも根拠を書く。なぜ良いのかの言語化が壁打ちの価値
